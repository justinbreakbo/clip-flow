# Android 后台同步产品路径

## 背景

Clip Flow 当前 Android 端已经有前台同步服务，可以通过 relay 长轮询接收远端内容，并尽力轮询本机文本剪贴板。

但 Android 10+ 对后台剪贴板访问有系统级限制：普通 App 即使运行前台服务，也不等于拥有后台剪贴板读写能力。产品上不能把“完全静默后台剪贴板同步”作为默认承诺，否则会在不同系统版本和厂商 ROM 上出现不稳定体验。

因此 Android 端后续按三条路径推进：

```text
默认体验：通知确认写入
增强体验：无障碍增强模式
长期高级体验：极简输入法模式
```

## 路径 1：通知确认写入

### 定位

这是 Android 端默认产品体验，当前已开始实现。

当 Android App 在后台收到来自 Windows 或其他设备的内容时，不默认强行静默写入系统剪贴板，而是保存为 pending clipboard，并发一条高优先级通知：

```text
来自 Windows PC 的剪贴板
点击复制到手机剪贴板
```

用户点击通知后，优先由后台服务直接写入 Android 剪贴板，不跳转到 App；如果系统限制后台写入，再 fallback 到打开确认页或轻量 Activity。

App 启动时会默认启动同步服务，并检查 `Clip Flow Clipboard Alerts` 通知渠道是否开启为高优先级。如果通知总权限关闭或渠道被设为静默，App 会提示用户跳转到系统通知设置。

### 适用场景

- App 不在前台时收到远端内容。
- 系统限制后台写入剪贴板。
- 内容被策略判断为敏感，例如密码、验证码、API Key。
- 用户开启“安全模式”或“手动确认模式”。

### 优点

- 最符合 Android 隐私边界。
- 上架风险最低。
- 用户能清楚知道什么时候有远端内容进入手机。
- 可以自然承载敏感内容确认。
- 实现成本较低，能最快改善当前后台同步体验。

### 缺点

- 不是完全无感的 Universal Clipboard。
- 用户需要多点一次通知。
- 如果用户忽略通知，手机剪贴板不会更新。

### 产品原则

- 默认启用。
- 普通文本可以提供“自动写入，失败时通知确认”的选项。
- 敏感内容始终默认通知确认。
- 通知内容默认只显示来源和类型；是否显示明文预览应由用户配置。

## 路径 2：无障碍增强模式

### 定位

这是可选的高级实验功能，用于改善 Android 后台和跨 App 场景下的自动同步体验。

无障碍模式不是默认开启能力，必须由用户在系统设置中手动授权。Clip Flow 只在用户明确开启后使用它。

### 可能能力

- 监听当前是否处在可输入场景。
- 在用户进入输入框时提示有远端剪贴板可用。
- 提供一键粘贴当前远端内容。
- 在部分场景下辅助执行粘贴动作。
- 改善 App 后台时的状态感知和提示时机。

### 优点

- 不要求用户更换默认键盘。
- 比普通前台服务更了解用户当前输入上下文。
- 可以把体验从“通知确认”推进到“进入输入框时自然提示”。

### 缺点

- 权限高度敏感，系统会显示强警告。
- Google Play 上架风险较高，必须明确说明用途和边界。
- 不应被用作暗中读屏、暗中点击、暗中读取敏感内容的通道。
- 不能保证所有 ROM 上都能稳定绕过剪贴板限制。

### 产品原则

- 不默认开启。
- 标记为“实验 / 高级 / 需要系统授权”。
- 开启前必须解释用途、风险和可关闭方式。
- 不暗中读屏，不采集无关窗口内容。
- 只围绕剪贴板同步、输入框提示和用户主动粘贴做最小化处理。

## 路径 3：极简输入法模式

### 定位

这是长期高级体验，适合做成独立模块：`Clip Flow Keyboard`。

因为 Android 系统允许默认输入法在输入场景中访问剪贴板，输入法模式是技术上最正统、最稳定的后台剪贴板体验路径。

### 可能能力

- 键盘候选栏显示最近远端剪贴板。
- 一键把来自 Windows 的内容粘贴到当前输入框。
- 显示最近剪贴板列表。
- 对敏感内容显示确认按钮，而不是直接粘贴。
- 在密码字段、银行 App、密码管理器等场景中默认隐藏或禁用同步建议。

### 优点

- 系统语义最合理。
- 在输入场景中稳定性最好。
- 能形成接近 Universal Clipboard 的自然体验。
- 市场上已有类似路径，例如 SwiftKey 这类输入法天然适合承载剪贴板和跨设备输入体验。

### 缺点

- 用户必须信任并启用 Clip Flow 作为输入法。
- 输入法是高信任入口，隐私说明和本地处理边界必须非常清楚。
- 开发量明显大于通知确认和无障碍增强。
- 不能一开始就试图替代成熟键盘，应先做极简同步键盘。

### 产品原则

- 作为独立高级模块推进。
- 不急着做完整输入法竞争。
- 第一版只做候选栏、远端剪贴板、一键粘贴。
- 不记录普通输入内容。
- 密码字段和敏感 App 默认禁用剪贴板建议。

## 推荐顺序

### 第一阶段：通知确认写入

目标是快速补齐后台体验：

- 后台收到远端内容后发高优先级通知。（已实现第一版）
- 点击通知后直接复制到 Android 剪贴板，不跳转 App。（已实现第一版）
- App 启动时检查通知权限和提醒渠道，必要时跳转系统设置。（已实现第一版）
- 敏感内容始终确认。
- App 内增加待处理远端剪贴板列表。
- 服务日志持久化，方便判断“没收到”还是“收到但没写入”。

### 第二阶段：无障碍增强模式

目标是提升输入场景体验：

- 增加可选无障碍服务。
- 引导用户手动授权。
- 进入输入框时提示有远端剪贴板可用。
- 支持用户主动一键粘贴。
- 明确标注实验功能和隐私边界。

### 第三阶段：极简输入法模式

目标是建立长期稳定入口：

- 增加 Clip Flow Keyboard。
- 在候选栏展示远端剪贴板。
- 支持一键粘贴和最近剪贴板。
- 建立敏感场景黑名单。
- 与现有前台服务、通知确认模式共存。

## 市场参照

### KDE Connect

KDE Connect 提供共享剪贴板、通知同步、文件分享等跨设备能力，是 Clip Flow 当前方向的重要参照。它证明跨设备剪贴板有真实需求，但在 Android 新版本上也会受到系统剪贴板权限变化影响。

### Microsoft Phone Link / Link to Windows

Phone Link 是 Windows 与 Android 的官方生态方案，包含跨设备复制粘贴能力。它依赖 Microsoft 账号、Windows 系统集成，以及部分手机厂商的预装或特殊角色。普通第三方 App 很难获得同等系统待遇。

### Microsoft SwiftKey

SwiftKey 是输入法模式的代表。输入法天然处于输入链路中，适合承载剪贴板候选、云剪贴板和跨设备输入体验。但这要求用户把产品设为默认键盘，信任门槛很高。

## 当前决策

Clip Flow Android 端采用以下产品路径：

```text
通知确认写入作为默认体验
无障碍增强模式作为可选实验功能
输入法模式作为长期高级模块
```

短期不追求“所有后台场景完全静默写入剪贴板”。产品应优先做到可靠、可解释、可控，再逐步增强自动化程度。
